# 当前状态与风险

- 更新时间：2026-03-28
- 适用范围：项目评估、排期、风险沟通、接手判断

## 1. 当前健康度

- 单元测试：通过
- CI：存在，覆盖 Python `3.9` 和 `3.12`
- 依赖复杂度：低
- 仓库结构：清晰，脚本职责分层基本明确
- 配置模型：可维护，支持覆盖式扩展

整体判断：这是一个“能维护、能继续迭代”的脚本型项目，不是一个失控仓库。

## 2. 当前最值得注意的风险

### 风险一：外部说明和代码事实已出现漂移

现象：
- README 写 `151` 个数据源
- 代码实际是 `167`

原因：
- 默认配置演进了，但项目说明没有同步刷新

影响：
- 产品对外承诺、测试预期、人工排查判断都可能基于旧数字，造成认知偏差

处理建议：
- 后续每次改 `config/defaults/` 后，同步更新 README 和 `docs/memory-bank/project-facts.md`

### 风险二：真实外部接口没有进入 CI

现象：
- 当前 CI 只跑离线单元测试

原因：
- 外部接口需要密钥、网络、限流，难以稳定纳入公共 CI

影响：
- Twitter、Web Search、GitHub 限流、RSS 结构变化等问题，可能在合并后才暴露

处理建议：
- 保留当前离线测试作为底线
- 关键改动后增加一次本地 smoke test 或手工联调

### 风险三：核心价值集中在合并与评分逻辑

现象：
- `merge-sources.py` 负责去重、分数、主题归属

原因：
- 项目真正影响日报质量的，不是“能抓到”，而是“如何筛”

影响：
- 这部分改坏了，表面上脚本仍成功执行，但产品结果会明显变差

处理建议：
- 涉及评分、去重、主题归组时，必须先补测试，再改逻辑

## 3. 当前开发优先级建议

如果接下来要继续投入开发，建议优先级如下：

1. 修正文档漂移，让 README 与代码事实一致
2. 补充运行与运维说明，降低接手成本
3. 给关键脚本增加更清晰的输出契约说明
4. 视需求补联调 smoke test 或更细的夹具测试

## 4. 本次初始化后的约束

- `memory-bank/` 记录当前事实，不放计划
- `guide/` 放流程和操作规范
- `product/` 放业务定位和目标用户
- `reports/` 放阶段性分析、复盘和一次性输出
- `plans/` 只放进行中的计划，完成后要沉淀到 `reports/`
