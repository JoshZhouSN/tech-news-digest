# 当前状态与风险

- 更新时间：2026-03-31
- 适用范围：项目评估、排期、风险沟通、接手判断

## 1. 当前健康度

- 单元测试：通过，当前基线 `67` 项
- CI：存在，覆盖 Python `3.9` 和 `3.12`
- 依赖复杂度：低
- 仓库结构：清晰，脚本职责分层明确

整体判断：这是一个“结构稳定、文档可落地、适合继续迭代”的脚本型项目。

## 2. 当前最值得注意的风险

### 风险一：外部接口没有进入公共 CI

现象：

- 当前 CI 只跑离线单元测试

原因：

- X、网页搜索、GitHub 等真实接口依赖密钥、本机登录态、网络和限流，难以稳定放进公共 CI

影响：

- 代码层面看起来是绿的，但真实运行时仍可能因为凭证失效、平台限流或返回结构变化而掉量

处理建议：

- 保留当前离线测试作为底线
- 关键改动后增加一次本地 smoke test
- 涉及 Bird 或 XCrawl 的变更，至少手工跑一次单层验证

### 风险二：后端选择逻辑容易被误解

现象：

- 很容易把 `auto` 理解成“什么都自动兜底”

原因：

- X 层和 Web 层的策略并不一样
- Bird 现在既可显式启用，也可能作为 auto 的最后兜底
- Web 显式指定后端时，不会跨后端偷偷切换

影响：

- 用户以为 auto 会覆盖所有后端，实际仍受“API 优先，Bird 最后兜底，且要求本机登录态可用”的约束
- 用户以为自己固定了 Brave，失败后会换 XCrawl，结果实际上只拿到 interface 输出

处理建议：

- 所有对外说明都坚持写清楚“auto 的真实顺序”和“explicit 强制指定的差别”
- 排障时先看 `backend` 或 `api_used` 字段，再讨论数据量异常

### 风险三：真正决定结果质量的是合并层

现象：

- `merge-sources.py` 负责打分、去重、域名限额、主题归组

原因：

- 项目真正影响日报质量的，不是“抓到了没有”，而是“抓到以后怎么筛”

影响：

- 这层改坏了，脚本表面仍然能成功执行，但用户看到的日报会明显变差

处理建议：

- 涉及评分、去重、主题归组时，必须先补测试，再改逻辑

## 3. 当前开发优先级建议

1. 保持 README、`.env.example` 和 `docs/` 与代码同步
2. 给真实外部接口准备更轻量的本地 smoke test 流程
3. 继续把关键脚本的输入输出契约写清楚，降低接手成本
4. 如果后续继续扩展搜索层，先明确“auto 是否要变更”再动实现

- `guide/` 放流程和运行规范

- `reports/` 放阶段性分析、实验和复盘
