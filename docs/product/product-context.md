# 产品背景

- 更新时间：2026-03-28
- 适用范围：产品判断、优先级讨论、功能取舍

## 1. 这个项目卖的是什么价值

它卖的不是“多抓几个网站”，而是“把分散、噪音多、更新快的科技信息，压缩成一份可消费的日报”。

换句话说：

- 对用户：减少信息筛选时间
- 对团队：把资讯监控流程标准化
- 对渠道：支持 Discord、邮件、PDF 等分发方式

## 2. 目标用户

当前从项目说明和默认配置可以推断，目标用户主要是：

- 想持续跟踪 AI、加密、前沿技术动态的个人用户
- 需要给群组、频道、团队定时推送摘要的运营或研究人员
- 希望通过对话式安装快速接入的 OpenClaw 用户

## 3. 产品核心竞争点

- 数据源覆盖面广
- 支持不同来源统一收敛
- 有质量评分与去重，不是简单拼接
- 支持多种输出形式
- 对安装和配置的要求相对低

## 4. 当前不是重点的方向

基于现有仓库结构，以下方向现在不像核心：

- 做成复杂 Web 后台
- 做实时流式监控平台
- 深度编辑器型的人工运营系统

如果后续要往这些方向走，架构和文档都需要明显升级，而不是只在脚本层继续堆功能。

## 5. 后续做决策时的取舍原则

### 优先做

- 提升日报质量的能力
- 降低接入和配置成本的能力
- 降低外部依赖不稳定带来损失的能力

### 谨慎做

- 只增加复杂度、不明显提升摘要质量的功能
- 需要大量新依赖的改造
- 会把脚本型项目推成高维护成本服务端系统的设计
