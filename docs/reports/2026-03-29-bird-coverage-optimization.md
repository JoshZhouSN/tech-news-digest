# Bird Coverage Optimization Report

- 更新时间：2026-03-29
- 适用范围：Bird 后端的覆盖率调优、运行参数选择、后续抓取策略讨论

## 1. 这次要解决什么问题

Bird 已经能读 X，并且输出格式和现有 Twitter 采集层兼容，但一旦跑完整默认源，很快就会撞上 `429`。这会让后半段账号几乎全军覆没，导致 Bird 只能做小样本验证，难以承担更大范围的补抓任务。

这次优化的目标不是“让 Bird 更快”，而是“在允许更长执行时间的前提下，尽量多抓到账号”。

## 2. 这次做了什么

### 改动一：把 Bird 节奏控制参数化

新增了 6 个 Bird 专用运行参数：

- `BIRD_MAX_WORKERS`
- `BIRD_REQUEST_INTERVAL_SEC`
- `BIRD_BATCH_SIZE`
- `BIRD_BATCH_COOLDOWN_SEC`
- `BIRD_429_COOLDOWN_SEC`
- `BIRD_MAX_CONSECUTIVE_429`

这样后续调参不需要再改代码常量，可以直接通过环境变量做实验。

### 改动二：Bird 改为顺序抓取 + 批次冷却 + 429 保护

Bird 现在不是把账号一口气并发丢出去，而是：

1. 一个账号一个账号按顺序抓
2. 每个账号抓完后等待固定间隔
3. 每跑完一批账号后，再额外冷却一段时间
4. 如果连续遇到 `429`，先冷却
5. 如果连续 `429` 达到阈值，直接触发 Bird 限流保护，停止继续硬撞

这意味着后半段账号不会再被无意义地反复冲撞限流。

## 3. 实验结果

### 基线一：Bird 并发 3，窗口 168h

- 成功账号：`14 / 48`
- 抓到文章：`61`
- 耗时：约 `151s`
- 主要问题：从第 15 个账号开始，后面大面积 `429`

### 基线二：Bird 串行，窗口 24h

- 成功账号：`25 / 48`
- 抓到文章：`26`
- 耗时：约 `385s`
- 第一个 `429`：第 26 个账号 `@herbertcrypto`

### 本次策略：Bird 串行 + 账号间隔 + 批次冷却 + 429 冷却 + 连续 429 保护，窗口 24h

- 成功账号：`25 / 48`
- 有文章账号：`9`
- 抓到文章：`24`
- 耗时：约 `551s`
- 第一个 `429`：第 26 个账号 `@herbertcrypto`
- 连续 `429` 到第 28 个账号后，触发 Bird 限流保护

## 4. 结果怎么理解

### 有改善的地方

- 这次策略成功把“后半段账号继续无意义撞 `429`”改成了“识别到会话已被限流后，主动停下来”
- 对平台更克制，也更适合后续做断点续跑或分批重试

### 没达到预期的地方

- 覆盖率没有比“纯串行 24h”更高，仍然停在 `25 / 48`
- 总耗时明显更长，从约 `385s` 增加到约 `551s`
- 这说明单纯加冷却和保护，还不足以把 Bird 的会话额度推到更后面

## 5. 当前建议

Bird 目前更适合：

- 本机补抓
- 少量高价值账号
- 分批跑，而不是一次吞 48 个默认源

Bird 目前不适合：

- 直接替代完整 Twitter API 路径
- 作为全量默认源的一次性主采集后端

下一步如果要继续提高覆盖率，最值得尝试的不是继续拉长单轮执行时间，而是：

1. 按主题或账号价值把默认源拆成多批
2. 在多批之间加入更长冷却
3. 做断点续跑，避免一轮限流后全部从头开始
